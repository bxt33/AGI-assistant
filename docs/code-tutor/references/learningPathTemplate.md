## 实习生学习路径模板

根据项目复杂度和实习生目标，生成个性化学习计划。

---

### 第一周：建立全局认知（目标：能画出架构图）

| 天数 | 目标 | 具体行动 | 验证标准 | 🏆 里程碑 |
|-----|------|---------|---------|-----------|
| Day 1 | 能跑起来 | 克隆项目、装依赖、本地启动 | 服务能正常响应请求 | 🎉 "Hello World" 成就 |
| Day 2 | 看懂全景 | 阅读入口文件、画出服务拓扑图 | 能说出项目有几个服务、各干什么 | 🗺️ "地图绘制者" 成就 |
| Day 3 | 摸清分层 | 阅读一个核心服务的目录结构 | 能说出每层的职责 | 📐 "分层大师" 成就 |
| Day 4 | 跑通链路 | 从一个 API 请求追到数据库 | 能画出一条完整调用链 | 🔗 "链路追踪者" 成就 |
| Day 5 | 理解配置 | 阅读配置文件、理解关键配置项 | 能说出改哪个配置会影响什么行为 | ⚙️ "配置达人" 成就 |

**第一周实操任务**：
- [ ] 本地启动服务，用 curl 或 Postman 调一个接口
- [ ] 画一张手绘架构图（不用很精确，大概对就行）
- [ ] 找到一个接口的入口函数，追踪 3 层调用

### 第二周：能独立改小需求（目标：提交第一个 MR）

| 天数 | 目标 | 具体行动 | 验证标准 | 🏆 里程碑 |
|-----|------|---------|---------|-----------|
| Day 6-7 | 精读核心模块 | 深入阅读 1-2 个核心模块的代码 | 能解释核心函数的逻辑 | 📖 "深度阅读者" 成就 |
| Day 8-9 | 改小需求 | 接一个简单需求，独立完成开发 | 代码能通过 CR | ✍️ "第一行代码" 成就 |
| Day 10 | 写测试 | 为自己的代码写单元测试 | 测试覆盖率达标 | 🧪 "质量守护者" 成就 |
| Day 11-12 | 理解上下游 | 了解项目的上下游依赖服务 | 能说出"我改了这里，谁会受影响" | 🌐 "全局视野" 成就 |
| Day 13-14 | 输出文档 | 写一份自己的理解笔记 | 能给下一个新人讲清楚 | 📝 "知识传承者" 成就 |

**第二周实操任务**：
- [ ] 照着已有的实现，模仿写一个类似的功能
- [ ] 为自己写的代码补充单元测试
- [ ] 写一份"我理解的项目架构"笔记（给未来的自己看）

### 按项目类型的重点补充

#### Clean Architecture 项目（Python / FastAPI）
- **重点理解**：四层依赖方向（domain → infrastructure → application → interfaces）
- **先看**：config.yaml → main.py → domain 接口 → infrastructure 实现 → application 编排 → interfaces HTTP
- **关键概念**：dataclass、ABC/Protocol、依赖注入、分层边界
- **实操**：追踪一次 `/api/chat` 从 handler 到 UnifiedAgent 的完整流程

#### Agent / AI 应用项目
- **重点理解**：四种路由模式（Chat/Tool/RAG/ReAct）、三层记忆、RAG 混合搜索
- **先看**：core_agent.py（UnifiedAgent） → rag/engine.py → memory/ → promptctx/assembler.py
- **关键概念**：Schema 驱动提示词组装、LLM 调用链、向量搜索、知识图谱
- **实操**：发送一条消息，追踪从 HTTP → Route → Process → LLM → Response 的完整链路

#### 大仓项目
- **重点理解**：共享库的作用、服务间的代码复用关系
- **先看**：根目录 README → 共享库（lib/）→ 核心服务
- **关键概念**：模块边界、版本管理、影响范围
- **实操**：找到一个共享库的函数，看看有哪些服务在用它

#### UDF/插件化架构
- **重点理解**：框架与扩展的关系、注册机制、接口约定
- **先看**：接口定义 → 注册入口 → 一个最简单的已有实现
- **关键概念**：Handler 接口、Try/Confirm/Cancel 生命周期、参数传递
- **实操**：照着最简单的已有 UDF，模仿写一个空壳（只返回固定值）

#### 配置驱动项目
- **重点理解**：哪些行为是配置决定的、配置的优先级
- **先看**：配置文件 → 配置加载代码 → 使用配置的业务代码
- **关键概念**：动态配置 vs 静态配置、配置热更新、七彩石
- **实操**：改一个配置项，观察服务行为的变化

### 学习效果自检清单

每完成一个阶段，用以下问题自检：

#### 🗺️ 全局认知自检（Day 1-5）
- [ ] 我能用一句话说清这个项目做什么
- [ ] 我能画出项目的架构图（不看代码）
- [ ] 我知道项目有哪些核心服务/模块
- [ ] 我知道一个请求从进来到返回经过了哪几步
- [ ] 我知道项目用了什么技术栈（语言、框架、存储）

#### 📐 模块理解自检（Day 6-7）
- [ ] 我能说出这个模块的职责边界
- [ ] 我知道谁调用它、它调用谁
- [ ] 我能找到这个模块的入口函数
- [ ] 我知道改这个模块会影响什么
- [ ] 我能解释这个模块为什么要这样设计

#### ✍️ 改代码能力自检（Day 8-14）
- [ ] 我知道加一个新接口需要改哪些文件
- [ ] 我知道加一个新业务逻辑从哪里入手
- [ ] 我能写出符合项目规范的代码
- [ ] 我能为自己的代码写单元测试
- [ ] 我能评估自己的改动对其他模块的影响

### 遇到困难时的求助策略

| 困难程度 | 建议行动 | 预计耗时 |
|---------|---------|---------|
| 😐 看不懂某个函数 | 先看函数注释和调用方，再问 AI | 5-10 min |
| 🤔 不理解设计意图 | 查 git log 看提交记录，或问 AI "为什么这样设计" | 10-15 min |
| 😣 环境跑不起来 | 先看 README，再查内部文档，最后问同事 | 15-30 min |
| 😰 改了代码不确定对不对 | 先跑单测，再找导师 CR | 10 min |
| 😵 完全迷失方向 | 回到全局架构图，重新定位自己在哪 | 5 min |

### 💡 高效学习小贴士

1. **不要试图一次看懂所有代码** — 先看主干，细节用到再看
2. **多画图** — 画出来的东西比脑子里想的清晰 10 倍
3. **带着问题看代码** — "这个请求怎么处理的？"比"我要看完这个文件"有效得多
4. **善用 git log** — 看提交记录能理解代码的演进历史
5. **模仿是最好的学习** — 找到一个类似的已有实现，照着改
6. **不要怕问问题** — 问"蠢问题"比闷头看 3 小时效率高得多
7. **写笔记** — 今天理解的东西，明天可能就忘了
