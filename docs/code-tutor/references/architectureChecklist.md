## 架构讲解检查清单（实习生版）

讲解项目架构时，按此清单逐项确认，确保实习生能建立完整认知。

### 1. 项目定位（先回答"这是什么"）
- [ ] 一句话说清项目做什么、服务谁
- [ ] 说明项目在团队/公司中的位置（上下游关系）
- [ ] 标注项目的核心业务场景（2-3 个）
- [ ] 解释"为什么需要这个项目"（业务价值）

### 2. 业务理解（先懂业务再看代码）
- [ ] 列出核心业务场景（用户视角）
- [ ] 用「用户故事」描述最常见的操作流程
- [ ] 解释关键业务术语（领域语言）
- [ ] 标注"哪些是核心流程，哪些是辅助流程"
- [ ] 画出业务流程图（不涉及代码，纯业务视角）

### 3. 架构模式识别
- [ ] 判断项目类型：微服务 / 大仓 / 单体 / 前后端分离
- [ ] 画出整体架构图（服务拓扑 / 模块关系）
- [ ] 标注服务间通信方式（RPC/HTTP/MQ/共享DB）
- [ ] 解释"为什么选择这种架构"（设计意图）

### 4. 入口与启动
- [ ] 找到项目入口（如 `main.py`、`app.py`、`main.go`）
- [ ] 说明启动时做了哪些初始化（配置加载、平台连接、中间件挂载）
- [ ] 标注"改了启动配置需要重启"还是"热加载"
- [ ] 给出本地启动的最简步骤

### 5. 分层结构
- [ ] 识别分层模式（如 domain → infrastructure → application → interfaces）
- [ ] 用类比解释每层职责
- [ ] 画出分层依赖图，标注调用方向
- [ ] 强调"哪些层可以跨层调用，哪些不行"（Clean Architecture：内层不依赖外层）
- [ ] 给出一个"从入口到底层"的完整示例

### 6. 核心模块（按重要性排序）
- [ ] 识别 3-5 个核心模块
- [ ] 说明每个模块的职责边界（一句话）
- [ ] 标注模块间的依赖关系
- [ ] 区分"⭐必看"和"用到再看"
- [ ] 每个模块给一个生活化类比

### 7. 数据流
- [ ] 说明数据从哪里来（API/数据库/缓存/消息队列）
- [ ] 画出一条典型请求的完整数据流转路径
- [ ] 标注数据在每一跳的变化（入参→出参）
- [ ] 标注"哪些数据是持久化的，哪些是临时的"

### 8. 配置与环境
- [ ] 找到配置文件（如 `config.yaml`、`.env`、`settings.py`）
- [ ] 解释关键配置项的作用（用大白话）
- [ ] 区分"本地开发配置"和"线上配置"
- [ ] 提醒哪些配置改错了会出大问题
- [ ] 说明配置的优先级（环境变量 > 配置文件 > 默认值）

### 9. 共享库与工具
- [ ] 识别项目中的共享库/公共组件
- [ ] 说明共享库被哪些服务依赖
- [ ] 提醒"改共享库要注意影响范围"
- [ ] 标注"哪些是项目自己的库，哪些是外部依赖"

### 10. 扩展点（实习生最关心的）
- [ ] "如果要加新接口，改哪里"（给出具体步骤）
- [ ] "如果要加新业务逻辑，从哪入手"
- [ ] "如果要接入新的外部服务，怎么加"
- [ ] 给出具体的文件路径和步骤
- [ ] **指出一个最简单的已有实现作为模板**

### 11. 阅读顺序推荐
- [ ] 给出"第一天看什么"的具体建议
- [ ] 标注每个文件的阅读优先级（⭐⭐⭐/⭐⭐/⭐）
- [ ] 估算每个文件的阅读时间
- [ ] 给出"看完后怎么验证自己理解了"的方法

### 12. 实操验证（新增）
- [ ] 给出一个"动手试试"的小任务
- [ ] 任务难度适中（30分钟内能完成）
- [ ] 完成任务后能验证对架构的理解
- [ ] 示例："找到 XXX 接口的入口，追踪到数据库操作，画出调用链"
