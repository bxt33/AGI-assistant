## 💡 它是什么？

**项目架构导读助手**是一位专为实习生设计的 AI 带教导师，采用 **「四层渐进」** 策略帮你系统理解复杂项目：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   🦅 鸟瞰层      🏢 业务层      🔍 模块层      💻 代码层                    │
│   全局认知地图 → 业务场景理解 → 单模块深潜  → 具体实现细节                  │
│   "项目做什么"   "业务怎么跑"   "模块干什么"   "代码怎么写"                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 核心设计理念

> 实习生的困境不是"看不懂语法"，而是 **"不知道从哪看起"** 和 **"看了局部拼不出全貌"**。

所以本助手遵循：
- **先全局后局部**：先给你项目地图，再放大细节
- **先业务后代码**：先讲业务目标，再看技术实现
- **先主干后分支**：先讲核心逻辑，再补充边界情况
- **先能用再精通**：先让你能上手，再追求深入理解

## 🏗️ 特别擅长的项目类型

| 项目类型 | 特殊策略 |
|:---|:---|
| **Clean Architecture（Python/FastAPI）** | 先看 domain 层接口定义，再追踪从 API 到数据库的完整链路 |
| **Agent / AI 应用** | 理解四种路由模式（Chat/Tool/RAG/ReAct），追踪 Schema 驱动提示词组装 |
| **微服务（tRPC-Go）** | 先看 proto 文件（接口契约），再画服务拓扑图 |
| **大仓 Monorepo** | 区分共享库和独立服务，画代码复用关系图 |
| **配置驱动** | 先解释配置文件，区分"改配置"vs"改代码" |

## ⚠️ 注意事项

- 你给的上下文越具体（文件路径、业务场景、你的目标），输出越精准
- 每次回答至少带 **1 张 Mermaid 图**
- 复杂项目会分层讲解，不会一次性全部倾倒
- 每次讲完都有"下一步选项"，跟着走就行
- ⚠️ 请确保项目在当前工作区内，否则无法扫描代码
- ⚠️ 若涉及敏感信息（密钥、内部域名），会提醒注意安全
- 💡 **永远不会让你觉得"问了个蠢问题"**，所有问题都值得认真回答
